## Problem Statement

Agam Arora is a 12-year AI product leader targeting senior PM roles at Anthropic, Stripe, Linear, Figma. No tier-1 pedigree (no FAANG, no Stanford/MIT). Career path: data analytics → game studio founder → beauty brand expansion → logistics platform 10x transformation → AI product leadership (RAG, Agentic AI, MAS, Voice AI, Copilots).

The current site and GitHub README read like every other PM resume — bullet points with impact numbers buried in generic formatting. Two prior redesign concepts were explored and rejected:
- **The Self-Constructing Portfolio** — page types itself into existence. Rejected: "shallow on interaction, narrative, AND meaning." 60 seconds of passive watching with no emotional arc.
- **The Playable Portfolio** — Phaser.js pixel art world. Prototyped and rejected: "a game mechanic without a game."

The core problem: Agam's career, read as a resume, looks like chaos — 5 unrelated industries in 12 years. The same career, experienced as a scroll-driven narrative, reveals convergence — every chapter built a skill that compounds into the current AI work. **The medium must change what the content means.**

## What Makes This Cool

Five visual threads — SVG paths representing career skills — start scattered across the screen and physically converge into a single point as you scroll. Behind the threads, full-page chapter sections tell the story of each career stage. The visitor controls the pace. The visual metaphor does the arguing before the text even kicks in.

A recruiter scrolls through and SEES the pattern emerge. They don't read "versatile." They watch five disconnected lines become one. The convergence is visual, kinetic, and undeniable.

Two separate layouts serve landscape (16:9) and portrait (9:16) screens. Same story, same content, completely different visual execution. No responsive compromises.

## Constraints

- **Two layouts:** Landscape (width > height) and portrait (height > width). Aspect ratio check at load + resize. Each layout has its own responsive range for minor stretch (e.g., landscape covers 16:9 through 21:9; portrait covers 9:16 through 3:4 iPad). Crossover: `if (viewportWidth > viewportHeight)` → landscape, else → portrait.
- Dark-first (Linear aesthetic). Light mode supported.
- Must load fast: <2s first paint, scroll interactive within 3s
- Total page weight budget: ~300KB target. Breakdown: Astro runtime (~10KB), GSAP + ScrollTrigger (~45KB gzipped), one font family (~30KB woff2), SVG paths (~5KB), CSS (~15KB), HTML + JS (~30KB). Leaves ~165KB headroom for content and images.
- Scroll must be 60fps on 2023 mid-range Android (e.g., Pixel 7a). See Performance Fallbacks section.
- Netlify deployment (existing pipeline)

### Accessibility

- `prefers-reduced-motion: reduce` → all content appears immediately in final state. Static convergence diagram (all 5 threads shown converged). No animation whatsoever.
- Semantic HTML: proper heading hierarchy (`h1` = name, `h2` = chapter titles, `h3` = subsections), `<main>`, `<section>`, `<nav>` landmarks.
- Skip-to-content link for keyboard users.
- All text in the DOM (not canvas-only). Screen readers read the full career narrative without needing scroll interaction.
- ARIA labels on SVG convergence paths: `aria-hidden="true"` (decorative, the text tells the story).
- No-JS fallback: all content visible as a static page. The animation is progressive enhancement on top of semantic HTML.
- Keyboard navigation: Tab through sections, Enter to activate CTA links. Scroll-triggered animations also advance with keyboard-driven scroll.
- Color contrast: all text meets WCAG AA (4.5:1 for body, 3:1 for large text) in both dark and light themes.

## Approaches Considered

### Approach A: "The Convergence Engine" (Visual Threads Only)
Five SVG paths start scattered, physically converge as you scroll. Text panels scroll alongside. The visual IS the argument.
- Tech: Vanilla HTML + GSAP ScrollTrigger + SVG
- Effort: M | Risk: Medium (SVG path math + mobile adaptation)
- Pros: Most novel. Visual metaphor does the arguing. Lightweight.
- Cons: SVG convergence on mobile is hard. Design ceiling limited without build tools. Visual might overshadow content.

### Approach B: "The Convergence Narrative" (Chapter Scroll)
Full-page scroll sections, one per career chapter. Each section has its own visual world. GSAP ScrollTrigger pins + transitions. Impact metrics animate on entry. Chapters end with "skill carried forward" bridge. Final section: five skills collapse into one.
- Tech: Astro + GSAP + Vite
- Effort: L | Risk: Low-Medium
- Pros: Proven scrollytelling pattern. Easier mobile. Content-first. Build tools enable polish.
- Cons: Less visually novel. Could feel like "nice Squarespace" without strong visual identity.

### Approach C: "The Convergence Engine + Narrative" (Hybrid) — CHOSEN
Approach A's visual threads as a persistent background layer. Approach B's chapter sections as the content layer scrolling on top. Threads converge behind the text as you read. Both layers synchronized to scroll position.
- Tech: Astro + GSAP + SVG + Vite
- Effort: L-XL | Risk: Medium-High
- Pros: Medium and message are inseparable. Visual threads prove convergence while text proves impact.
- Cons: Two synchronized animation layers. Mobile requires separate layout. Highest complexity.

## Recommended Approach

**Approach C: The Hybrid**, built iteratively. Start with Approach A (SVG convergence + text panels) as the skeleton. Layer in Approach B's richer chapter content once the convergence animation works. This de-risks the hardest part (SVG path animation) first.

**Mobile strategy:** Two completely separate layouts for landscape (width > height) and portrait (height > width). Same story, same content, different visual execution. On portrait, the SVG convergence threads become a convergence progress indicator (five circles that merge as you scroll). The argument stays; the rendering changes. **Note:** dual-layout roughly doubles the animation engineering work. Mitigated by phasing: Phase 1 ships landscape only. Portrait is Phase 2, and can ship as a simpler version (static convergence diagram + scrolling text cards) if full animation proves too costly.

**Tech stack:**
- Astro — static-first, minimal JS, perfect Netlify fit, component islands for interactive sections
- GSAP ScrollTrigger — scroll-to-animation mapping, section pinning, progress tracking
- SVG — convergence paths, animated via `stroke-dashoffset` + GSAP-interpolated control points
- Vite (via Astro) — CSS nesting, font optimization, dev server

## The Experience — Scroll Model

**Scroll mechanic:** The entire page is one tall document. Total height = ~7.5x viewport height (7 full-height sections at ~100vh each, plus a shorter contact section at ~50vh). GSAP ScrollTrigger creates a single timeline mapped to total scroll progress (0 at top, 1 at bottom). Each section is a `<section>` element. The SVG convergence layer is `position: fixed` on the viewport and reads from the same ScrollTrigger progress value.

**Orientation change:** On device rotation (landscape <-> portrait), the page reloads to the appropriate layout. Current scroll percentage is saved to `sessionStorage` before reload and restored after, mapped to the new layout's total height. This avoids the complexity of animating between two completely different visual layers at runtime.

**Scroll direction handling:** Scrolling backward reverses the convergence — threads diverge back. This is natural because GSAP ScrollTrigger's `scrub` mode ties animation to scroll position, not scroll direction. At any scroll position, the visual state is deterministic.

**Repeat visitors:** On second visit (detected via `localStorage`), offer a "Skip to portfolio" link in the top-right corner that smooth-scrolls to Section 5 (AI / current work). First-time visitors see no skip option.

### Landscape Layout (width > height)

**Background layer (persistent, `position: fixed`):** Five SVG cubic bezier paths. Default stroke-width: 1.5px. Default opacity: 0.4. Muted color palette (TBD, see Open Question 4). Starting positions as viewport percentages:
- Thread 1 (Analytics): starts at (5vw, 15vh)
- Thread 2 (Gaming): starts at (8vw, 75vh)
- Thread 3 (Beauty): starts at (90vw, 20vh)
- Thread 4 (Logistics): starts at (85vw, 80vh)
- Thread 5 (AI): starts at (50vw, 5vh)
- Convergence point: (75vw, 50vh) — right of center to create asymmetry and leave the left 60% of the viewport clear for text panels in odd-numbered sections

Each path has two independent animation properties:
1. **Visibility (draw-on):** Controlled by `stroke-dasharray`/`stroke-dashoffset`. Each thread becomes visible when its chapter enters the viewport (e.g., Thread 1 draws on at scroll 12%, Thread 4 at 55%). Once drawn, the thread stays visible.
2. **Convergence (position):** Controlled by interpolating bezier control points. ALL threads move toward the convergence point simultaneously across the full scroll range (progress 0->1), but threads that haven't drawn on yet are invisible. The effect: as you scroll, each thread appears at its chapter and is already partially converged based on scroll position.

GSAP `scrub: true` drives both properties from the same scroll progress value.

## Content Architecture Per Chapter

Each chapter follows this structure:

```
DOMAIN — one word/phrase
THE HONEST CHALLENGE — what you walked into (1-2 sentences)
THE SPECIFIC DECISION — what you actually did, specific enough to be credible (2-3 sentences)
THE HARD NUMBER — one metric that proves it
THE SKILL CARRIED FORWARD — one sentence connecting to AI convergence
```

**Content is TBD.** Agam needs to write the specific decisions and honest challenges for each chapter. The framework is locked. The words are not.

**Content length constraints per field:**
- DOMAIN: 1-3 words
- THE HONEST CHALLENGE: 1-2 sentences, max ~150 characters
- THE SPECIFIC DECISION: 2-3 sentences, max ~300 characters
- THE HARD NUMBER: one metric, max ~50 characters
- THE SKILL CARRIED FORWARD: one sentence, max ~100 characters

**Overflow behavior:** Each section is exactly 100vh. Text panel has a max-height of ~60vh with `overflow-y: auto` as a safety valve. Content should be written to fit without scrolling — the character limits above are calibrated for a 600px-wide panel at 18px font.

**Build vs. content dependency:** The technical skeleton (Astro project, GSAP scroll setup, SVG convergence animation, layout switching) can be built with placeholder text. Content authoring and technical implementation can proceed in parallel. The content slots into the existing framework without changing the architecture.

## Performance Fallbacks

The hardest rendering challenge: 5 animated SVG paths as a `position: fixed` layer with foreground content scrolling on top, all at 60fps.

**Primary approach:** SVG paths with `will-change: transform` on the fixed container. GSAP `scrub: true` interpolates control points. SVG repaints are limited to path `d` attribute changes, not full reflows.

**Fallback 1 (if SVG jank on mobile):** Replace SVG paths with a single `<canvas>` element. Pre-calculate path interpolation for 100 scroll positions (1% increments). On scroll, draw the nearest pre-calculated frame. Canvas repaints are cheaper than SVG DOM mutations.

**Fallback 2 (if canvas still janky on low-end devices):** Reduce to 3 threads (cut Analytics and Beauty chapters from the visual, keep them in text). Fewer animated paths = less rendering work. The convergence metaphor still works with 3 threads.

**Fallback 3 (nuclear):** `prefers-reduced-motion` or detected low-end device -> static convergence diagram + scrolling text. No animation. Still a great portfolio.

**Testing:** Profile on Chrome DevTools Performance tab with 4x CPU throttle. Target: no frames >16ms in the scroll timeline. **Escalation is a build-time decision made during Phase 1:** if the SVG approach drops more than 5% of frames on a throttled Pixel 7a profile, switch to canvas for all devices. The fallback is not runtime-adaptive.

## Landscape Research

Searched 2026-03-28. Key findings:
- PM portfolios are still boring — clean Squarespace sites with resume content, even from Stanford/Harvard PMs
- Scrollytelling is proven (SCMP, The Pudding, BMW) but nobody applies it to PM career narratives
- GSAP ScrollTrigger is the industry standard. Working Stiff Films designed their entire site as "one continuous storyline" using it.
- Companies are dropping degree requirements and evaluating through portfolio reviews — investment in portfolio quality is directionally correct
- Key insight: "user-driven pacing creates emotional investment — when people control the story's progression, they become participants rather than observers"
- The gap: everyone uses scrollytelling to make content more engaging. Nobody uses it to reveal a pattern that the reader couldn't see from raw data. The convergence scroll doesn't just present the career better — it changes what the career means.
