## 1. Refined statement

**Lean Startup canonical loop applied as PM operating principle: build the smallest set of features that delivers customer value, measure with feedback, iterate.** Foundational/root belief — the methodology floor under almost every other PM-craft belief. Held since 2016, never superseded, but has been *layered on top of* by later beliefs (idea-validation-3-pillars, ship-the-prototype, value-viability-usability-triad, spec-over-sprint).

The belief is not "build any MVP" — it is specifically the **build → measure → learn loop** as the operating cadence of product work. By 2024 Agam adds an explicit layer: BML alone is no longer enough; "Building and measuring products is not difficult now. Learning to delight is the next pitstop." (2024-03-23). Methodology floor → methodology floor + delight ceiling.

**Key refinement (proposed):** The belief is best stated today as *"BML is the operating loop, but the loop is necessary-not-sufficient — it has to be paired with taste, delight, and (post-2026) spec-first context."*

## 3. Counter-evidence

Where does Agam **depart from** strict BML?

### Counter-evidence 3.1 — "Learning to delight is the next pitstop" (2024-03-23)
The single strongest internal critique. BML is *necessary but not sufficient* — by 2024, Agam considers BML "easy" and the harder problem is delight. This isn't a contradiction; it's an upgrade. But it is a departure from BML-as-complete-methodology.

### Counter-evidence 3.2 — Anti-customization (2024-12-04)
A pure-iterate-everything BML reading would let users vote (via measurement) on customization features. Agam refuses pre-emptively. This is a **spec-discipline** layer that overrides the iterate-everything reading. Not contradiction — domain restriction.

### Counter-evidence 3.3 — Spec-over-sprint (2026-04-09)
"When you've already spent hours speccing every pixel... a generative tool gives you a worse version of what you've already decided." This is anti-iteration in a narrow sense — Agam is saying once-the-spec-is-locked, iteration with generative tools degrades the work. The full reconciliation: BML happens at the *product* loop level (build-measure-learn what users want); spec-over-sprint operates at the *generation* level (don't let an LLM iterate on something you've already converged).

### Counter-evidence 3.4 — "10 experiments → 2 production grade wins" (2024-09-04)
Implicitly anti-BML at the portfolio level. If 8 of 10 experiments don't graduate, the BML loop measured them as not-valuable. The portfolio-level pattern says enterprise BML is **filtering, not just iterating** — most experiments die. Not a refutation; a sober statement of how the loop ends.

### Counter-evidence 3.5 — None contradict the core BML claim
Notably, **no post in the corpus rejects BML**. There is no waterfall-defense post, no spec-then-build refutation. The closest critique is "delight is the next pitstop" — which is *additive*, not subtractive.

### TENSION CANDIDATE: `belief.spec-over-sprint`
The brief's headline question: BML = iterate-then-spec; spec-over-sprint = spec-then-iterate. Reconcile.

**Reconciliation:** these operate at **different altitudes** and **different scopes**. They are not contradictory.

| Axis | BML | spec-over-sprint |
|---|---|---|
| **Scope** | Whole product strategy / feature decisions | Single artifact generation (UI, code, design output) |
| **Question answered** | "What should we build?" | "How do we get to the spec already decided?" |
| **Loop participants** | Product + customer + data | PM + tool (LLM, generator) |
| **When in lifecycle** | Pre-PMF, also at every feature decision | Inside an already-validated execution |
| **Direction of iteration** | Build → Measure → Learn → Build (open-ended) | Spec → Generate → Filter → Spec-tighten (converging) |

The 2026-04-09 spec-over-sprint post is *specifically about generative tools degrading already-locked design tokens*. Agam's hours-speccing-every-pixel happens **after** the BML loop has converged on what to build; spec-over-sprint kicks in at the generation-execution stage. **BML still runs at the strategy altitude** — Agam still ships prototypes (the Aug 2025 voice-AI demos, the second-brain wiki, the Sep 2024 enterprise GenAI POCs are all BML evidence at AIonOS). What's superseded by spec-over-sprint is the **prior `belief.ship-fast`** stance (2020 era) — *speed at the expense of taste* is gone; BML *with discipline + spec* remains.

**Proposed ontology framing:** No tension. Two beliefs at adjacent altitudes both held simultaneously. Add an edge `belief.spec-over-sprint --constrains-execution-of--> belief.build-measure-learn`.

### Other relation edges
